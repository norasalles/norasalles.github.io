Why Sync your Sabre Profiles with ClientBase?

Security

  • All data is protected according to PCI and GDPR standards

Efficiency

  • ClientBase and Sabre Profiles contain the same client profiles, making ClientBase information available to your after-hours service and any advisors who don’t use ClientBase
  • Edits made in one system will sync to the other, reducing duplicate work
  • Use ClientBase’s superior search capabilities to manage your Sabre profiles
  • Use ClientBase’s Global Modify feature to edit profiles in bulk, then sync the changes to Sabre Profiles
  • Sabre Profiles can be marketed to using ClientBase tools: ClientBase Marketing Service, Marketing Codes, Merge to E-mail and Merge to Letter

Structure

  • Using our Sabre Profiles templates with your Group, Corporate, and Traveler Sabre Profiles ensures that all information entered is pertinent to ClientBase

Flexibility

  • Access your ClientBase profiles in Sabre Profiles and vice versa, whichever is convenient for you

What information will sync?

  • Sabre Group, Corporate, and Traveler Profiles will sync to ClientBase. (Other profile types do not sync.)
  • ClientBase Leisure and Corporate profiles will from to Sabre Profiles, along with the Travelers that are associated to them.
  • A Sabre Profiles Corporate Profile corresponds to a ClientBase Corporate Profile.,

What Happens to my Data?

Initial Sync In

Before there can be ongoing synchronization between Sabre Profiles and ClientBase, an initial Sync In must be performed. This is a one-time import which brings Sabre Profiles into ClientBase by merging and inserting Profiles and Travelers.

The initial import process finds all the Group, Traveler, and Corporate profiles in a given PCC. (Purged profiles are ignored.) It then makes a list of each of those profiles' Sabre Profile Index number (SPUniqueIDs) and sends that list to be processed as part of the ongoing sync to ClientBase. We first attempt to match Sabre Profiles with ClientBase Profiles and Travelers.

  • If we can find a matching profile or traveler in ClientBase for a Sabre Profile, those records will be merged.
  • If no match can be found, the Sabre Profile or Traveler will be inserted.

Profile Matching Logic

We only compare Sabre Profiles Group profiles to ClientBase Leisure Profiles. We only compare Sabre Profiles Corporate Profiles to ClientBase Corporate Profiles.

If a Sabre Profile has a DK number or Account Number that corresponds to a ClientBase profile’s Interface ID, that is considered a match. The profiles will be merged together.

Traveler Matching Logic

We start by looking at the numbers we know to be unique for a given traveler: Known Traveler Number, Redress Number, and Passport Number (together with the issuing country).

  • If the Sabre Profiles traveler and the ClientBase traveler have the same Known Traveler Number, that is a unique number, so if they match, we merge the two together into one traveler.
  • If the Sabre Profiles traveler and the ClientBase traveler have the same Redress number, same reasoning as above. That is a unique number, so we merge the two travelers into one.
  • If the Sabre Profiles traveler and the ClientBase traveler have the same Passport number and the same issuing country, that is a unique match, so we merge the two travelers together.

If none of the above conditions are true, then we look at the Traveler names and birthdates. First Name, Middle Name, Last Name, and birth date must all match. Exception: if one traveler has a middle name and the other does not, this can be considered a match if all the other criteria match.

What happens during a Profile Merge?

When a Sabre Profiles Profile is merged into a matching ClientBase Profile, if the corresponding ClientBase field is empty and the Sabre Profiles field contains data, we populate the ClientBase field with the Sabre Profiles data. Similarly, if the corresponding Sabre Profiles field is empty and the ClientBase field contains data, the Sabre Profiles field is populated with the ClientBase data. If both contain data, then whichever record was updated more recently will update the other side.

This rule applies when there is only one record allowed for the type, for example, Known Traveler Number, Name fields, and Birth Date fields. When more than one record of a given type is allowed, for example, phone numbers, payment cards, and passports, we attempt to match up those records and update them, but if we cannot, we will insert as an additional record.

What Else Happens During Initial Sync In?

Sabre Profiles have a unique identifying number called the Sabre Profile Index. When a Sabre Profile is brought into ClientBase, that number is attached to that profile in ClientBase. Once those profiles are linked together via that number, they are paired. Updates to one are synced to the other.

Also, each record within the ClientBase profile is numbered. For example, phone number 1, 2, and 3. The corresponding phone numbers in Sabre Profiles are assigned the same numbering – 1, 2, and 3. From then on, an update to the phone number 1 in either side updates the other phone number 1. This way we don’t have to continually search for matches every time sync runs. 

During the Sync In process, if a profile in ClientBase is found via one of the methods above, the Profiles are merged. That means that if a ClientBase field is empty and the corresponding Sabre Profile field has data, the ClientBase field is populated with Sabre Profile data.

If a ClientBase field and the corresponding Sabre Profile field BOTH have data, then the date and time the ClientBase record was last modified and the date/time the Sabre Profile was last modified are compared. The field with the most recent update time will update the other.

Some areas of the profile are not as straightforward. Let’s look at some of the other things we do to align Sabre Profiles data with ClientBase data.

Address Types

ClientBase and Sabre Profiles both assign Address Types, but the organization is a bit different. Importantly, Sabre Profiles has no concept of a Primary address, but ClientBase does. Here’s how we assign ClientBase Address types when a Sabre Profile is synced in. These rules apply both to merged and inserted profiles.

Establishing the ClientBase Profile’s Primary Address

  • If there is already a Primary address in ClientBase, that one remains marked as Primary.
  • If there is no Primary address in ClientBase, or if the Primary address is BLANK, the first Sabre Billing Address will be marked as the Primary in ClientBase.
  • If the Sabre Profiles profile does not have any Billing addresses, then the first Delivery Address will be marked as the Primary.
  • If the Sabre Profiles profile does not have any Delivery addresses, the first remaining Address will be marked as Primary

Assigning Other Address Types

  • The first Billing Address will be inserted as the Bill To address.
  • The first Delivery Address will become the Ship To address.
  • Any additional Billing, Delivery, or uncategorized addresses will be inserted into ClientBase without a type.
  • If there is only one address in Sabre Profiles, and no address in ClientBase, it is inserted as one address with the Primary, Bill To, and Ship To boxes checked.
  • If two or more Sabre Profiles addresses are the same, we insert it into ClientBase only once and check all the applicable address type boxes. For example, if there are two Sabre Profile addresses, but one is Billing and one is Delivery, we will insert them as one address with both Bill To and Ship To checked in ClientBase.
  • If there is already a ClientBase Bill To or Ship To address, we do not replace it with the Sabre Profiles Billing or Delivery address. UNLESS the Bill To or Ship To are BLANK., This would mean if there is an address in ClientBase marked as Primary, Bill To, and Ship To, as is the default, it keeps all its designations.

Communications

  • A Primary email address is only designated when the ClientBase profile or traveler has no existing email addresses.
  • The first Sabre Profiles email address will be set as Primary. Other email addresses are inserted without a type.
  • If an email address does not exist in ClientBase that does exist in Sabre Profiles, the email address will be added to the ClientBase profile.
  • If an email address does not exist in Sabre Profiles that does exist in the ClientBase profile, the email address will be added to the Sabre Profiles profile.
  • Phone/Fax: In Sabre Profiles, Fax is a type of phone number. In ClientBase, Fax is a separate communications type. If a Sabre Profiles Phone Type is Fax, then the ClientBase type will be Fax. If the Sabre Profiles Phone Type is anything else, then the ClientBase type will be Phone.
  • If a phone number is found in ClientBase that matches one in Sabre Profiles, then whichever one has been updated more recently will update the other. If there is no match found for the phone number, If a phone number is found in ClientBase that matches one in Sabre Profiles, then whichever one has been updated more recently will update the other. If there is no match found for the phone number, it will be inserted as a new phone number.
  • Note: If the ClientBase communication type is Fax; then it will only be compared to Fax communications in Sabre Profiles.
  • Sabre Profiles phone numbers of ‘Agency’ type will not be synced to ClientBase.
  •  
  • Sabre Profiles Phone Numbers are converted to the format [+1 (212) 555-1212] before being inserted or merged.
  • Primary phone is only designated when the ClientBase Profile or Traveler has no phone numbers. Primary is then set as the first Sabre Profiles phone number.

Emergency Contact

  • If an emergency contact does not exist in ClientBase that does exist in Sabre Profiles, the emergency contact will be added to the ClientBase Traveler.
  • If the same emergency contact exists in both travelers (First Name and Last Name match), then if the Sabre Profile record is more recently updated, the ClientBase traveler will be updated. If the ClientBase traveler is more recently updated, then the Sabre Profiles traveler will be updated.
  • If the ClientBase emergency contact does not have both a first and last name PLUS a phone number and/or email address, it will not sync to Sabre Profiles, as it will not meet the Sabre Profiles requirements for an emergency contact.

Passports

  • If a passport does not exist in ClientBase that does exist in Sabre Profiles, the passport will be added to the ClientBase Traveler.
  • If a passport does not exist in Sabre Profiles that does exist in the ClientBase traveler, the passport will be added to the Sabre Profiles traveler.
  • If the same passport exists in both travelers (Passport number + Issuing Country match), then if the Sabre Profiles Traveler record is more recently updated, the ClientBase traveler will be updated.
  • If the ClientBase traveler is more recently updated, then the Sabre Profiles traveler will be updated.

Payment Cards and Loyalty Cards

  • If a Payment card is found in ClientBase that matches one in Sabre Profiles (Card Code and Card Number must match), then whichever one has been updated more recently will update the other. If there is no match found for the card, it will be inserted as a new card.
  • When a Loyalty Card is found in a ClientBase Traveler that matches a card in the Sabre Profiles Traveler, the two cards will be sync-linked (mapped to each other, or matched for syncing). To be considered a match, the Sabre Profiles Vendor Type Code must match the ClientBase Card Type, the Sabre Profiles Vendor Code must match the ClientBase Card Code, and the Sabre Profiles Membership ID must match the ClientBase Card Number. If a match is found, then whichever record was modified most recently will update the matching record in the other system. If no match is found, then the Loyalty Card will be added as a new card.

Ongoing Sync
Once sync has been established, updates to In Sabre Profiles, what we consider an "update" is actually the deletion of the old record and the addition of the new record. You will see this if you examine the Profile Update History in Sabre Profiles.

  • If there is a deletion and an addition of a record in the Sabre Profile, then the corresponding record is updated in ClientBase.
  • If there is only an addition of a record in the Sabre Profile, then an update is performed in ClientBase.
  • If there is only a deletion in the Sabre Profile, then a delete is performed in ClientBase. 
  • If there is no match and there is only an Add action in the Sabre Profile, then an insert is performed in ClientBase.
  • If there is no match and there is only a Delete action in the Sabre Profile, then nothing is changed in ClientBase.
  • If there is a match on Delete, then no match on Add, in ClientBase the matching record is deleted, and the new record (with the new address from the Sabre Profile) is inserted in ClientBase

Import Rules

If a Profile in ClientBase is found via a lookup method that does not use the UniqueID, the Profile is merged and logged to the changelog.

If a matching ClientBase field is empty/null, and the Sabre Profile element has data, the ClientBase field is populated with the Sabre Profile Data.

What Happens During Ongoing Sync?
After the initial sync in has completed, an ongoing sync process is begun to update and insert Profiles. The sync routine first looks to find the Sabre Profile ID in ClientBase. This is a unique Sabre Profiles Profile number that has come over in the initial sync in process.

In Sabre Profiles, what we would consider an "update" is actually the deletion of the old record and the addition of the new record.

  • If there is a matching record for both a Delete action and an Add action in the Sabre Profile, then an update is performed on that record in ClientBase.
  • If there is only a matching record on an Add in the Sabre Profile, then an update is performed in ClientBase.
  • If there is only a match on a Delete in the Sabre Profile, then a delete is performed in ClientBase. 
  • If there is no match and there is both a Delete action and an Add action in the Sabre Profile, then an insert is performed in ClientBase.
  • If there is no match and there is only an Add action in the Sabre Profile, then an insert is performed in ClientBase.
  • If there is no match and there is only a Delete action in the Sabre Profile, then nothing is changed in ClientBase.
  • If there is a match on Delete, then no match on Add, in ClientBase the matching record is deleted, and the new record (with the new address from the Sabre Profile) is inserted in ClientBase

Import Rules

  • If a Profile in ClientBase is found via a lookup method that does not use the UniqueID, the Profile is merged and logged to the changelog.
  • If a matching ClientBase field is empty/null, and the Sabre Profile element has data, the ClientBase field is populated with the Sabre Profile Data.
  •  

Merge Rules

  • If a matching ClientBase field is empty/null and the Sabre Profile field has data, the ClientBase Field is populated with Sabre Profile data.
  • If a matching ClientBase field and a Sabre Profile field BOTH have data, the Data/Time the record was last modified is compated to the UpdateDateTime of the Sabre Profile. If both are equal, the Sabre Profiles Profile wins.
  • No half merges! For each subject area, if a matching record is found in ClientBase, ALL of the Sabre Profiles record will be merged. (This means blank fields from the Sabre Profiles record may replace populated ClientBase fields in these areas.

NOTE: Fields that only exist in one product (e.g. Marketing Permission in ClientBase) are ignored and will never be overwritten or updated.
