Sabre Profiles Synchronization

Overview

The Sabre Profile Synchronization routine provides a two-way sync between Sabre Profiles and ClientBase Profiles. Because Sabre Profiles and ClientBase Profiles differ in structure, this document references:

 

  • What Syncs?
  • What Profile Fields Match up?
  • Wrapper Profile Creation
  • What Constitutes a Match During Initial Sync-In?
  • What Constitutes a Match/Update During On-Going Sync?
  • What Data Fields Are Deleted vs. Updated?
  • Update Logic

 

What Syncs?

Sabre Profiles and ClientBase Profiles do not have similar program designs. For example, while a Sabre Profiles Corporate Profile is synonymous with a ClientBase Corporate Profile, a Sabre Profiles Corporation Profile can have multiple names and multiple AN/DK Numbers, while the current ClientBase Corporate Profile design allows for only one name and one AN/DK Number. Therefore, only one name and one AN/DK Number is imported, and the rest is ignored.

 

Likewise, a Sabre Profiles Group Profile is synonymous with a ClientBase Leisure Profile. While a Sabre Group Profile can have can multiple names and AN/DK Numbers, the current ClientBase Leisure Profile design allows for only one name and one AN/DK Number. Therefore, only one is imported and the rest is ignored.

 

A Sabre Profiles Traveler Profile is synonymous with a ClientBase Traveler/Contact within a Profile. This is another design difference as the ClientBase Traveler/Contact must be linked to a ClientBase Profile and in the Sabre Profiles, the Traveler Profile is not. Furthermore, a Sabre Profiles Traveler can have multiple names, while the current ClientBase design allows for only one name. Therefore, only one name is imported and the rest is ignored. If a Sabre Profiles Traveler Profile does not have any valid associations to any other ClientBase Profiles, a new “wrapper” ClientBase Profile is created.

If a Sabre Profiles Traveler Profile does not have any valid associations to any other ClientBase Profiles, a new Corporate “wrapper” ClientBase Profile is created. What Constitutes a Match During the Initial Sync-In?

Before there can be on-going synchronization between Sabre Profiles and ClientBase, an initial Sync-In must be performed. We define this as Syncing in all data from Sabre Profiles into ClientBase for an agency with no previous Sabre Profiles information. This is a one-time import which brings Sabre Profiles into ClientBase by merging and inserting Profiles and Profile data. During this process, the first consideration is to determine what constitutes a match between Sabre Profiles and ClientBase Profile data fields.

What Matches During Initial Sync-In?

During the Sync-In process, if a Profile in ClientBase is found via a lookup method during the Initial Sync that does not use the UniqueID, the Profile is merged and logged to the changelog. If a matching ClientBase field is empty/null and the Sabre Profile element/attribute has data, the ClientBase field is populated with Sabre Profile data.

 

If a matching ClientBase field and a Sabre Profile element/attribute BOTH have data, then the Date/Time the record was last modified and the UpdateDateTime of the Sabre Profile is compared. If both are equal, the Sabre Profiles Profile wins.

Determination

ADDRESS

 
  • No merging takes place for Addresses during an import, only updates.
  • For Traveler Addresses: the ClientBase Traveler Address is only replaced if the ClientBase Traveler does not have an address, but the Sabre Profiles Traveler does have an address. Insert the Sabre Profiles Traveler Address into the ClientBase Traveler.  Otherwise, do nothing.
  • For Profile Addresses: Address records are sorted by OrderSequenceNo value (non-visible attribute). Addresses with a Billing LocationTypeCode (BIL) are grouped together. Addresses with a Delivery LocationTypeCode (DEL) are grouped together.  Any remaining Address records are grouped together, regardless of the LocationTypeCode.
  • Rules for assigning Address Types in ClientBase.
    • If there is already a Primary address in ClientBase, keep that one marked as Primary.
    • If there is no Primary address in ClientBase, or if the Primary address is BLANK, use the Billing Address with the lowest OrderSequenceNo value (non-visible attribute). 
    • If the Sabre Profiles profile does not have any Billing addresses, use the Delivery Address with the lowest OrderSequenceNo value (non-visible attribute). 
    • If the Sabre Profiles profile does not have any Delivery addresses, use any remaining Address element with the earliest OrderSequenceNo value (non-visible attribute). 
    • If there is no Sabre Profiles Address data, do nothing.
    • The Billing Address with the earliest OrderSequenceNo value (non-visible attribute) will be inserted as the Bill To address.
    • The Delivery Address with the earliest OrderSequenceNo value (non-visible attribute) will become the Ship To address.
    • If two or more Sabre Profiles addresses are the same except for the LocationTypeCode, insert it into ClientBase only once and check all the applicable address type boxes. For example, if all the matching criteria are identical for two Sabre Profile address records, but one has LocationTypeCode = BIL and one has DEL, create one address with both Bill To and Ship To checked in ClientBase.
    • Any additional Addresses will be inserted into ClientBase as uncategorized. This means extra Billing, Delivery, and uncategorized Addresses with later OrderSequenceNos values (non-visible attributes.)
    • If there is only one address in Sabre Profiles, and no address in ClientBase, insert it as Primary/Bill To/Ship To.
    • If there is already a ClientBase Bill To/Ship To, do not replace this designation with the Sabre Profiles Billing or Delivery address. Insert as uncategorized UNLESS the Bill To and/or Ship To are BLANK, in which case, overwrite with the Billing or Delivery address with the earliest OrderSequenceNo value (non-visible attribute). This would mean if there is an address in ClientBase marked as primary/billto/shipto, as is the default, it keeps all its designations.
    • If there is no ClientBase "Bill To" (and the ClientBase Primary and "Ship To" are), insert the Sabre Profiles BILLING address with the earliest OrderSequenceNo value (non-visible attribute) as the ClientBase "Bill To". 
    • Similarly, if there is a defined ClientBase Primary and "Bill To" and no defined ClientBase "Ship To"; insert the Sabre Profiles DELIVERY address with the earliest as the ClientBase "Ship To".
  • ClientBase Address1 must match Sabre Profiles AddressLine 1
  • ClientBase Address2 must match Sabre Profiles AddressLine 2
  • ClientBase City must match Sabre Profiles City
  • ClientBase Zip Code must match Sabre Profiles Zip
  • ClientBase State must match Sabre Profiles code/description.  This means that the ClientBase State must match the Sabre Profiles State code exactly. Either the ClientBase State is converted to a Sabre Profiles State Code and matches the Sabre Profiles State Code, or the ClientBase State matches the Sabre Profiles State Code converted to its description.
  • ClientBase Country must match Sabre Profiles code/description.  This means that the ClientBase Country must match the Sabre Profiles Country code exactly. Either the ClientBase Country is converted to a Sabre Profiles Country code and matches the Sabre Profiles Country code, or the ClientBase Country matches the Sabre Profiles Country code converted to its description.
  • If Country is empty in ClientBase and State exists, default the Country as United States if the State is a valid US state.
  • Apt/Suite (which populates from Sabre Address Line 3) intentionally is not considered for matching purposes. Consider Apt 3, Number 3, #3, etc.

  • Email: No merging takes place for Email addresses during an import, only updates.
  • The Sabre Profiles and ClientBase Email Address must match. If no match is found, an insert will occur.
  • Primary email is only set when the ClientBase profile or traveler has no existing records of that type.  Primary is then set to the Sabre Profiles record with the earliest OrderSequenceNo value (non-visible attribute).  If the data exists for a type, primary is not changed for any existing or new record.
  • Phone/Fax: No merging takes place for phone/fax numbers during an import, only updates.
  • If no match is found, an insert will occur.
  • The converted communication types must match.  If the Sabre Profiles Phone Location Type Code is Fax, then the ClientBase type of the potential match must be Fax.
  • If the Sabre Profiles Phone Location Type Code is anything else, then the ClientBase type of the potential match must be phone.
  • Sabre Profiles Phone Number is converted to canonical format [+1 (212) 555-1212].  The converted Sabre Profiles field must match the ClientBase Field.
  • Primary phone is only set when the ClientBase Profile/Traveler has no existing records of that type.  Primary is then set to the Sabre Profiles record with the earliest OrderSequenceNo value (non-visible attribute).  If the data exists for a type, primary is not changed for any existing or new record.

EMERGENCY CONTACT
  • No merging takes place for Addresses during an import, only updates.
  • The ClientBase Traveler's Emergency Contact is only updated if the ClientBase Traveler does not have an emergency contact, but the Sabre Profiles Traveler does have an Emergency contact. Insert the Sabre Profiles Traveler Emergency Contact into the ClientBase Traveler.  Otherwise, do nothing.

PAYMENT CARDS
  • No merging takes place for Credit Cards during an import, only updates.
  • If no match is found, an insert will occur.
  • The Sabre Profiles Bank Card Vendor Code must match the ClientBase Card Code.
  • The Sabre Profiles Card Number must match the ClientBase Decrypted Card Number OR the Sabre Profiles Card Number, once masked using the ClientBase masking logic, must match the ClientBase Masked Card Number.

  • No merging takes place for Loyalty Cards during an import, only updates.
  • If no match is found, an insert will occur.
  • The converted Sabre Profiles Vendor Type Code must match the ClientBase Card Type.
  • The Sabre Profiles Vendor Code must match the ClientBase Card Code.
  • The Sabre Profiles Membership ID must match the ClientBase Decrypted Card Number.

PASSPORTS
  • Sabre Profiles Document number must match the ClientBase Passport No.

CLIENTBASE TRAVELER (fields of only 1 instance)
  • Field values must match exactly.
  • If a matching ClientBase field is empty/null and the Sabre Profiles element/attribute has data, populate the ClientBase Field with Sabre Profiles Data. Otherwise do nothing.

CLIENTBASE PROFILE (fields of only 1 instance)
  • Field values must match exactly.
  • If a matching ClientBase field is empty/null and the Sabre Profiles element/attribute has data, populate the ClientBase Field with Sabre Profiles Data. Otherwise do nothing.

What Constitutes a Match/Update During On-Going Sync?

After the initial Sync-In has completed, an On-Going sync process is begun to update and insert Profiles. The sync routine first looks to find the Sabre Profile ID in ClientBase. This is a unique Sabre Profiles Profile number that has come over in the intial sync in process. The Sabre PCC identifies the branch.

 

 

In Sabre Profiles, what we would consider an "update" is actually the deletion of the old record and the addition of the new record.

 

  • If there is a matching record for both a Delete action and an Add action in the Sabre Profile, then an update is performed on that record in ClientBase.
  • If there is only a matching record on an Add in the Sabre Profile, then an update is performed in ClientBase.
  • If there is only a match on a Delete in the Sabre Profile, then a delete is performed in ClientBase.
  • If there is no match and there is both a Delete action and an Add action in the Sabre Profile, then an insert is performed in ClientBase.
  • If there is no match and there is only an Add action in the Sabre Profile, then an insert is performed in ClientBase.
  • If there is no match and there is only a Delete action in the Sabre Profile, then nothing is changed in ClientBase.
  • If there is a match on Delete, then no match on Add, in ClientBase the matching record is deleted, and the new record (with the new address from the Sabre Profile) is inserted in ClientBase.

 

Import Rules

 

  • If a Profile in ClientBase is found via a lookup method that does not use the UniqueID, the Profile is merged and logged to the changelog.
  • If a matching ClientBase field is empty/null, and the Sabre Profile element has data, the ClientBase field is populated with the Sabre Profile Data.

 

Merge Rules

 

  • If a matching ClientBase field is empty/null and the Sabre Profile element/attribute has data, the ClientBase Field is populated with Sabre Profile data.
  • If a matching ClientBase field and a Sabre Profile element/attribute BOTH have data, the Data/Time the record was last modified is compated to the UpdateDateTime of the Sabre Profile. If both are equal, the Sabre Profiles Profile wins.

 

No half merges! For the following subject areas, if a matching record is found in ClientBase, ALL of the Sabre Profiles record will be merged. (This means blank fields from the Sabre Profiles record may replace populated ClientBase fields in these areas.

 

NOTE: Fields that only exist in one product (e.g. Credit Limit in ClientBase, etc., are ignored and will not cause such overwriting.

What Updates During On-Going Sync?

All updates are dictated by the ChangeHistory of the Sabre profile obtained by the ProfileHistoryRQ. The match logic occurs with Addresses (GRP/CRP only), Emails, Phone, Loyalty Cards (TVL only), Passports, and Credit Cards. The most recent ChangeHistory with a match is used, and all older ChangeHistory items are ignored.

 

Subject

Determination

Payment cards
  • Update the ClientBase credit card record with the Sabre Profile credit card record if the Date/Time the record was last modified is earlier than the Sabre Profiles UpdateDateTime. If no match is found, an insert will occur.

Loyalty cards
  • Update the ClientBase loyalty card record with the Sabre Profile loyalty card record if the Date/Time the record was last modified is earlier than the Sabre Profile UpdateDateTime. If no match is found, an insert will occur.

Passports
  • If the Date/Time the record was last modified of the Traveler record is earlier than the Sabre Profile UpdateDateTime, replace the two ClientBase passports with first 2 Sabre Profile passports. If there are fewer than 2 Sabre Profile Passports, then the ClientBase passport fields of the non-matching passport are cleared.

ADDRESSES
  • For TRAVELERS, if the Date/Time the record was last modified is earlier than the Sabre Profile UpdateDateTime, replace the ClientBase Traveler address with first Sabre Profile address (or clear the ClientBase Traveler address if the Sabre Profile Traveler has no addresses).  
  • For PROFILES, update matching addresses if the Date/Time the ClientBase address record was last modifed is earlier than the Sabre Profile UpdateDateTime.
  • If no matching address is found, the unmatched Sabre Profile addresses will be inserted into ClientBase as uncategorized.
  • If a ClientBase address is already marked Primary, do not replace the address with this designation with a non-matching address unless blank.
  • If there is no Primary address in ClientBase, or if the Primary address is BLANK, insert/update with the Sabre Profile Billing Address with the lowest OrderSequenceNo value (non-visible attribute).  
  • If there is already a ClientBase Bill To/Ship To, do not replace the address with this designation with a non-matching Sabre Profile Billing or non-matching Delivery address. Insert these as uncategorized.
  • If there is no Bill to address in ClientBase, or if the Bill To address is BLANK, insert/update with the Sabre Profile Billing Address with the lowest OrderSequenceNo value (non-visible attribute).  
  • If there is no Ship To address in ClientBase, or if the Ship To address is BLANK, insert/update with the Sabre Profile Billing Address with the lowest value.
  • This would mean if there is an address in ClientBase marked as Primary/Bill To/Ship To, as is our default, it keeps all its designations.
  • Any additional unmatched Sabre Profile Addresses of any Type will be inserted into ClientBase as uncategorized.
  • If there is only one address in Sabre Profiles, and no address in ClientBase, insert it as Primary/Bill To/Ship To.
  • If two or more Sabre Profile addresses are the same except for the LocationTypeCode, insert it into ClientBase only once and check all the applicable address type boxes. For example, if all the matching criteria in are identical for two Sabre Profile address records, but one has LocationTypeCode = BIL and one has DEL, create one address with both Bill To and Ship To checked in ClientBase.
  • If there is no ClientBase "Bill To" (and the ClientBase Primary and "Ship To" are non-matching), insert the Sabre Profile BILLING address with the lowest OrderSequenceNo value (non-visible attribute) as the ClientBase "Bill To".  
  • Similarly, if there is no ClientBase "Ship To" (and the ClientBase Primary and "Bill To" are non-matching); insert the Sabre Profile DELIVERY address with the lowest as the ClientBase "Ship To".

What Data Fields Are Deleted vs. Updated

Let’s explore when customer data is updated or deleted during the On-Going Sync Process.

Updating and Deleting ClientBase Traveler Data from Sabre Profiles

A ClientBase Traveler/Contact within a Profile is synonymous with a Sabre Profiles Traveler Profile. When a ClientBase Traveler is deleted from ClientBase, it will not be deleted from Sabre Profiles. This table describes how Sabre Profiles will respond to subject areas (e.g. Communications, Cards, Passports, etc.) in a ClientBase Traveler/Contact Profile being deleted.

 

Section

Determination

General
  • Changes to the General section indicate that user is updating a Traveler, not deleting.
  • Name data is contained in elements called PersonName. If the first PersonName element’s Courtesy Title, First Name, Last Name, Middle Name, and Salutation/Nickname do not match, the element will be deleted.
  • A new PersonName element will be created if any of the Subject Areas is not blank.
  • Gender, Citizenship, and Birthdate are in the Customer element. If any of the Subject areas do not match the matching attributes in Customer element is updated, these attributes will be updated in the Customer element.

Communications
  • If a ClientBase Traveler’s communication data is deleted or updated, all of the supported Sabre Profiles communications will be deleted and all of the supported ClientBase communications will be inserted. This is done because ClientBase does not maintain a history of the data in a Communication record just prior to update or deletion.
  • Phone and Fax records come from the same Sabre Profiles element, Telephone. Whether or not if a Telephone element becomes a phone or fax communication type in ClientBase is determined if the LocationTypeCode attribute is “FAX”.
  • Email records come from Email SP elements.

Address
  • If a ClientBase Traveler’s address data is deleted or updated, the first Sabre Profiles address will be deleted. If the ClientBase Traveler has an address, the address data will be re-inserted.
  • Address data is contained in elements called Address. The first Address element will be deleted. A new Address element will be created if any of the Subject Areas is not blank.

Passport
  • Passport: Passport data is contained in elements called Document with a DocTypeCode of “PSPT”. The elements are Passport Number, Expiration, Issue Date, Issuing city and Issuing Country. If the first two passport Document elements do not match, the elements will be deleted. New Document elements will be created if any of the Subject Areas are not blank.
  • Redress Number: Redress Number is in the Customer element. If the Redress Number does not match the attribute in Customer element, the attribute will be updated in the Customer element.
  • Known Traveler Number: Known Traveler Number is in the Customer element. If the Known Traveler Number does not match the attribute in Customer element, the attribute will be updated in the Customer element.
  • Emergency Contact (Name, Phone, Miscellaneous, Notes): If a ClientBase Traveler’s emergency contact data is deleted or updated, the first Sabre Profiles emergency contact will be deleted. If the ClientBase Traveler has an emergency contact, the emergency contact data will be re-inserted.
    • If the first EmergencyContactPerson element’s Name and Misc do not match or the Phone is not found in any of the Telephone elements, the element will be deleted. A new EmergencyContactPerson element will be created if any of the Subject Areas is not blank.
    • The Name field is the combination of the individual Sabre Profiles Name field elements in the following format: “FirstName MiddleName LastName” Example: John Michael Doe.
    • An emergency contact can have multiple phone numbers, so the Phone must match one of the phone numbers.

Cards
  • If a ClientBase Traveler’s card data is deleted or updated, all of the supported Sabre Profiles credit cards and non-credit cards will be deleted and all of the supported ClientBase cards will be inserted. This is done because ClientBase does not maintain a history of the data in a card record just prior to update or deletion.

Traveler Preferences
  • Traveler Preferences are not supported.

PNR Entries
  • PNR Entries are not supported.

Associated Profiles
  • Only Sabre Profile types of Corporation and Group are valid. All other profile types are ignored. If a delete occurs, all the Associated Profiles records with a UniqueID set (non-visible to the user) will be reviewed. If any valid links exist in the Sabre Profile where they do not exist in ClientBase, the Sabre Profile Associated Profile records will be deleted.
  • Sabre Profiles Corporation Profile type maps to ClientBase Corporate type. Sabre Proiles Group Profile type maps to ClientBase Leisure type. Updates are ignored. A ClientBase update to an Associated Profile link does not contain any data that can be updated.
  • Associated Profiles are looked up via AssocUniqueID and AssocProfileTypeCode attributes. The Associated Profile links on any Sabre Profiles Corporation and Group Profile types will be reviewed and deleted if needed.

Determination

General
  • Some name data is contained in elements called ContactName. If the first ContactName element’s Courtesy Title, First Name, Last Name, and Middle Name do not match, the element will be deleted.
  • A new ContactName element will be created if any of the Subject Areas is not blank.
  • Some name data is contained in elements called GroupInfo. If the first GroupInfo element’s GroupName does not match the Additional Name, the element will be deleted. A new GroupInfo element will be created if any of the Subject Areas is not blank.

Communications
  • If a Leisure Profile communication data is deleted or updated, all of supported Sabre Profiles communications will be deleted and all of the supported ClientBase communications will be inserted. This is done because ClientBase does not maintain a history of the data in a Communication record just prior to update or deletion.
  • Phone and Fax records come from the same Sabre Profiles element, Telephone. Whether or not if a Telephone element becomes a phone or fax communication type in ClientBase is determined if the LocationTypeCode attribute is “FAX”.
  • Email records come from Email Sabre Profiles elements.

Address
  • If any of a ClientBase Leisure Profile’s address data is deleted or updated, all of the corresponding Sabre Profile’s addresses will be deleted and all of the ClientBase addresses will be inserted. This is done because ClientBase does not maintain a history of the data in an Address record just prior to update or deletion.

Cards
  • If any of a ClientBase Leisure profile’s card data is deleted or updated, all of the supported Sabre Profiles credit cards and non-credit cards will be deleted and all of the supported ClientBase cards will be inserted. This is done because ClientBase does not maintain a history of the data in a card record just prior to update or deletion.
