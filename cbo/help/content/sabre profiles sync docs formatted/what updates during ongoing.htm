What Updates During Ongoing Sync?


All updates are dictated by the ChangeHistory of the Sabre profile obtained by the ProfileHistoryRQ. The match logic occurs with Addresses (GRP/CRP only), Emails, Phone, Loyalty Cards (TVL only), Passports, and Credit Cards. The most recent ChangeHistory with a match is used, and all older ChangeHistory items are ignored.

What happens when a profile is deleted?

When a ClientBase profile is deleted, upon syncing the change to Sabre Profiles, it is marked for purge and set to delete in 24 hours. If the Sabre Profile is updated after the ClientBase profile is updated, but before the change syncs over to Sabre Profiles, then the Sabre Profile will not be purged, and the updates made will sync the Profile back to ClientBase as a new profile. 

When a Sabre Profiles profile is marked for deletion, it is not deleted in ClientBase. Instead, the corresponding ClientBase profile’s PCC and Sabre Profiles Index Number are cleared. The profile remains in ClientBase as a non-syncing profile. If a purged SP profile is then restored after the CB profile is deleted, it will sync back to CB as a new profile.

Wrapper Profile Creation

If a Sabre Profiles Traveler Profile does not have any valid associations to any other ClientBase Profiles, a new Leisure “wrapper” ClientBase Profile is created. Fields imported are First Name, Last Name, Middle Name, and Courtesy Title. The Sabre Profiles Traveler Profile is linked into this Leisure “wrapper” ClientBase Profile. (If desired, this new Profile can be changed from Leisure to Corporate by using the Type drop-down in the General Info tab.)

These wrapper profiles do not sync to Sabre Profiles. They exist solely to contain standalone Sabre Profiles Travelers.

When will a ClientBase wrapper profile be created for a Sabre Traveler, and when will it not? Let’s look at the possible scenarios.

What happens to credit card data?

When handling your credit card data, we follow strict data security standards set forth by the Payment Card Industry (PCI). These standards are a part of the PCI Data Security Standards (PCI DSS) and any company that processes credit card payments is required to adhere to them.

• Credit cards in Sabre Profiles are required to have a Payment Usage Type. Sabre Profiles Sync will use “All” by default for all credit cards synced into Sabre Profiles from ClientBase.

 

Working with Sabre Profiles Templates and Sabre Profiles Sync

Templates are a feature of Sabre Profiles that makes it easier to ensure that the correct fields are completed. They can be used to hide parts of the Sabre Profile that you don’t need, to make fields required, and more.

We have created default templates for your use in Sabre Profiles. Corporate, Group, and Traveler profiles each have their own template. These templates expose the fields that are supported in Sabre Profiles Sync to ClientBase.

When a profile or traveler is imported into Sabre Profiles from ClientBase, if it doesn’t already have a template associated with it, then the corresponding default ClientBase template is applied.

If your agency already uses templates, or if you would prefer to create your own templates, that is perfectly fine. Your agency’s templates are still available to use when creating Sabre Profiles. We will only attach our default templates to those profiles that don’t already have one.

Setting up to use Sabre Profiles Synchronization

 

The INIT Report

Before setting up for an initial (INIT) sync, you’ll want to make sure that your customer profiles in both Sabre Profiles and ClientBase are in good shape. That means making sure that duplicates have been cleaned up, that outdated profiles (for example, test profiles) are purged, and so on. To help you locate these profiles, we created the INIT report. Essentially, we take your Sabre Profiles data and do a mock initial sync with your ClientBase Online data. Then we show on the report what would have happened if there had been an actual Initial Sync In.

 

Setting up Agency Branch Access with our ClientBase IPCC

An agency administrator needs to add branch access in SRW for each of their participating agency PCCs to our IPCC: 1LWF. Use this format to grant or delete branch access to other ARC/BSP-approved Sabre system or Abacus locations. This allows access to your agency records and customer profiles.

W/TA*B4T0/AU

AUTHORIZED PSEUDO CITIES

A0B0-BC A1I0-B A6E0-B B9K0-BC B9M0-B B9N0-B B9O0-B

C2O0-BC C5M0-B C8I0-B C9Z0-B D1I0 D2O0 F4X0-BC

Q7L0-B S1M0 Y5P0-BC Z1C0-B F790-BC F920 J430-BC

 

 

 

After you use this format, a "B " displays beside the pseudo city code (PCC) to which you granted access in your Travel Journal Record (TJR). B indicates that you granted branch access to an ARC/BSP-approved location.

  • A = Stands for “approved” (no indicator following TJR code) access and is not billable.
  • Pseudo city codes (PCC) without a B or BC indicator are non-ARC/BSP approved locations (A).
  • There is no check to see if the agency is ARC or BSP approved.
  • This authorization type of access provides access to agency records and customer profiles. (Only ‘A’ level is necessary for ‘S’ (Can Sign-In) level to be granted.)
  • B = Branch Access - B. Branch access is the same as “A” access except that it is also needed for:
  • Ticketing / View ticketing data from another PCC
  • AAA (BC) Branch Access – ‘B’ level must be given before ‘C’ (AAA) level can be granted. (Only ‘A’ level is necessary for ‘S’ (Can Sign-In) level to be granted.)
  • Invoice & Itinerary display - all PCCs in the PNR History Record
  • Agent Sine Security – ‘update the Sign-In Message
  • Certain pricing entries that use the “B” level check
  • Monitoring another PC (if appropriate authorization and EPR keywords are active)
  • C = Branch Access (BC) - (AAA capability).
  • Branch Access – ‘B’ level must be given before ‘C’ (Can AAA) level can be granted.
  • (Only ‘A’ level is necessary for ‘S’ (Can Sign-In) level to be granted.)
  • D = Branch Access D (BD).
  • Branch Access – ‘B’ and ‘C’ level must be given before ‘D’ level can be granted.
  • Branch Access D allows an agency to AAA into another agency in order to create, book, price and ticket PNRs. This setting inhibits the usage of the following keywords in the agent’s TJR: PNRREL, ACCESS, CREATE, CCVIEW and SUBMGR.
